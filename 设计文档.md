# Smallville God 设计文档

> 最后更新：v0.3（通用建筑系统 + 工具修理/制作 + 角色面板）

## 1. 项目概述

中世纪欧洲背景下，3 名移民在荒废小镇合作求生、抵御严冬的回合制模拟项目。玩家扮演"观察者"，不直接操控任何角色——所有决策由 LLM（Gemini）驱动的 NPC 自行做出。

核心矛盾：**有限的工具、资源、时间**，与**互补但性格迥异的 3 人团队**之间的协作取舍。

### 1.1 技术栈

- 构建：Vite 5（`npm run dev` / `build` / `preview`）
- 运行：浏览器端 ES Modules，纯前端无后端
- 渲染：DOM UI（无 Canvas）
- LLM：`@google/genai`（Gemini），配合本地规则引擎做降级/校验
- 测试：Vitest

### 1.2 核心理念

- **LLM 主导，规则兜底**：LLM 负责叙事和决策，规则引擎负责一致性校验与 LLM 不可用时的降级运行
- **数据驱动**：游戏内容（NPC、地点、技能、物品）全部定义在 `behaviorLibrary.json`；可调数值集中在 `config.js`
- **NPC 心智持久化**：每个 NPC 有独立的短/长期目标、记忆、约定，睡前通过 LLM 整理一天

## 3. NPC 行为框架

### 3.1 四层架构

```
层级1：需求系统（WHY）  —— NPC 此刻最想满足什么需求
         ↓ 驱动选择
层级2：行为库（WHAT）    —— 可执行行为的客观分类
         ↓ 约束于
层级3：能力集（CAN）     —— 每个 NPC 掌握的技能子集
         ↓ 约束于
层级4：环境（WHERE）     —— 地点 × 工具 × 材料决定能否执行
```

### 3.2 需求层：SURVIVE / PRODUCE / BOND / SEEK

```
                物质            精神
           ┌───────────┬───────────┐
  指向自身 │  SURVIVE   │   SEEK    │
           │  存续      │   求知    │
           ├───────────┼───────────┤
  指向外部 │  PRODUCE   │   BOND    │
           │  创造      │   联结    │
           └───────────┴───────────┘
```

每个 NPC 对四种需求有独立的权重与表达风格（`needs.*.weight` / `style`），这构成了角色的核心性格差异。

### 3.3 行为层：PHYSICAL / SOCIAL / MENTAL / RESTORE

- **PHYSICAL（劳作）**：作用于物质世界——采集、伐木、狩猎、烹饪、建造、修理工具
- **SOCIAL（交际）**：与他人双向交互——交易、协调、说服、安慰、协商、闲谈
- **MENTAL（思索）**：内在信息处理——筹划、观察
- **RESTORE（休养）**：自我恢复——睡眠、进食、歇息

### 3.4 3 NPC 精简阵容（互补设计）

游戏以"3 人协作生存"为核心张力，每个角色在技能、性格、需求权重上都构成必要的互补。单独任一角色都无法撑过严冬，必须相互配合。

| NPC | 年龄 | 关键技能 | 性格关键词 | 不擅长 |
|---|---|---|---|---|
| **Agnes（阿格尼丝）** | 35 | cook / forage / tend_fire / smoke_meat + 4 SOCIAL | 热情、果断、善于说服 | 无法砍树/打猎/修房 |
| **Roderic（罗德里克）** | 28 | hunt / craft_clothing / forage | 独立、沉默、技艺精湛 | 不擅社交、抗拒群居 |
| **Oskar（奥斯卡）** | 45 | chop_wood / build / gather_stone / haul / tend_fire / repair_tool / craft_tool | 固执、沉默、一丝不苟 | 不会打猎、不会做饭 |

协作必要性：
- 只有 Agnes 能把 Roderic 的生肉变成熟肉，让大家吃饱
- 只有 Oskar 能把石木茅草变成御寒的小屋
- Agnes 是唯一能让沉默的两人坐下来商量的纽带

## 4. 物品系统

### 4.1 消耗品（8 种）

| id | 中文 | 营养/燃料 | 来源 | 备注 |
|---|---|---|---|---|
| berries | 浆果 | 营养 3 | forage | |
| raw_meat | 生肉 | 营养 5 | hunt | 会腐烂（spoilDays=2） |
| cooked_meat | 熟肉 | 营养 8 | cook | |
| cured_meat | 腌肉 | 营养 6 | smoke_meat | 不腐烂 |
| timber | 木材 | 燃料 4 | chop_wood | 建材兼燃料 |
| stone | 石块 | — | gather_stone | |
| hide | 兽皮 | — | hunt | |
| thatch | 茅草 | — | forage | |

### 4.2 耐久品（6 种）

| id | 中文 | 最大耐久 | 备注 |
|---|---|---|---|
| axe | 斧头 | 10 | chop_wood 需要 |
| hammer | 铁锤 | 50 | build / gather_stone / repair_tool / craft_tool 需要 |
| bow | 弓 | 8 | hunt 需要；耐久有限，是狩猎链的瓶颈 |
| needle_thread | 针线 | 6 | craft_clothing 需要 |
| cooking_pot | 铁锅 | 15 | cook 需要 |
| fur_coat | 皮毛外套 | 20 | 御寒品；3 张兽皮缝制一件 |

### 4.3 工具修理与制作

工具耐久耗尽会彻底损坏消失。Oskar 可在营地（或建成后的工坊）使用两个新技能应对：

- **repair_tool**：石块×1 → 给最破旧的工具恢复 +5 耐久（工坊内 +8）
- **craft_tool**：按配方消耗材料，重新制作已损坏缺失的工具（工坊内 timber 材料减 1）

配方定义在 `config.TOOL_RECIPES`，优先级按 `axe → bow → needle_thread → cooking_pot → hammer` 顺序自动选择最急需的缺失工具。铁锤自身的修理/制作不需要另一把铁锤（避免死锁）。

## 5. 建筑与地点系统

### 5.1 10 个地点

**基础地点（4 个，开局即有）**：

| id | 中文 | 用途 |
|---|---|---|
| camp | 营地 | 做饭、看火、集会；大部分 RESTORE 行为发生地 |
| forest | 森林 | forage / hunt / chop_wood 的主要场所，冬季危险 |
| ruins | 废屋 | gather_stone 场所；修缮完成后即成为第一间"小屋"可避寒 |
| storehouse | 储藏室 | 物资总仓；plan / gossip 也常在这里 |

**可建造地点（6 个，开局为空地）**：见 5.3 节。

### 5.2 通用建筑项目系统

所有可建造的建筑共享同一套 **"备料 → 投入劳动"** 两阶段流程：

1. **备料阶段**：首次执行 build 时，从 storehouse / 项目地点 / camp / 执行者一次性扣除 `materials`
2. **劳动阶段**：每小时投入 1 点进度，累计达到 `laborHours` 即完工

定义在 `config.BUILDING_PROJECTS`；运行时状态保存在 `GameState.state.projects[id] = { progress, materialsDelivered }`。通过 `GameState.advanceProject(id)` / `deliverProjectMaterials(id)` / `isProjectComplete(id)` 等通用 API 操作。

### 5.3 7 个可建造项目

| id | 名称 | 材料 | 劳动小时 | 效果类型 | 具体效果 |
|---|---|---|---|---|---|
| ruins | 小屋（废屋修缮） | thatch×6 + stone×10 | 20 | 硬 | 首要目标，未修好则露天过冬 |
| house_agnes | 阿格尼丝的屋子 | timber×8 + thatch×4 | 12 | 硬 | 她在此睡眠时能量恢复 ×1.5 |
| house_roderic | 罗德里克的屋子 | timber×8 + thatch×4 | 12 | 硬 | 同上 |
| house_oskar | 奥斯卡的屋子 | timber×8 + thatch×4 | 12 | 硬 | 同上 |
| workshop | 工坊 | timber×10 + stone×4 | 12 | 硬 | Oskar 在此修理 +8 耐久，制作少 1 木材 |
| smokehouse | 烟熏房 | stone×6 + timber×4 | 10 | 硬 | 解锁 `smoke_meat`：生肉+木材 → 不腐烂的腌肉 |
| shrine | 神龛 | stone×8 + timber×4 | 15 | 软 | 注入 LLM 叙事上下文，强化团结/慰藉 |

资源压力设计：3 间民居 24 木材 + 工坊 10 + 烟熏房 4 + 神龛 4 = 42 木材，加上炉火燃料、工具制作消耗，构成"要不要建次要建筑"的真实取舍。

### 5.4 建筑效果查询 API

```
GameState.hasHouseSleepBonus(npcId)  →  返回该 NPC 的民居 id 或 null
GameState.isWorkshopBuilt()
GameState.isSmokehouseBuilt()
GameState.isShrineBuilt()
```

这些查询被 `BehaviorSystem` 和 `prompts.js` 消费，用于硬加成和软提示。

## 6. 时间与季节

### 6.1 Tick 模型

- 每个游戏小时 = 1 个 tick（默认现实 20 秒/tick，可 1×/2×/4×/8× 加速）
- 每个 tick 内：按地点分组 → 每组调用一次 LLM 或规则引擎决策 → 应用技能结果 → 扣需求 → 推进时间
- 夜间（22:00–05:00）自动触发集体睡眠；22 点做一次日终心智整理（MindSystem）

### 6.2 冬季四阶段

| 阶段 | 周 | 生存压力 | 关键特征 |
|---|---|---|---|
| 秋末 lateAutumn | 1–3 | 低 | 资源充足，备冬窗口 |
| 深秋 deepAutumn | 4–6 | 中 | 气温骤降，备冬紧迫 |
| 初冬 earlyWinter | 7–9 | 高 | 大雪封路，协作变得必要 |
| 隆冬 deepWinter | 10–12 | 极端 | 暴风雪、食物见底、疾病风险 |

压力等级通过 `ENV_TEMPERATURE` 放大饱食/体力消耗。

## 7. LLM 决策架构

### 7.1 同地点分组决策

同一地点的所有 NPC 共用一个 Prompt，由 LLM 以"导演"身份编排因果链（观察→对话→冲突→反应）。这让同地点的 NPC 可以自然产生对话、协商、分工，而不同地点的 NPC 互不干扰。

Prompt 构建见 `llm/prompts.js::buildGroupPrompt`，包含：
- 当前时间/冬季阶段/地点物资
- 仓库物资
- **【建筑项目】**：所有项目的状态（未开工/备料完成/建造中/已建成），LLM 据此决定下一个目标
- **【生存形势】**：按需触发的硬提示（废屋未修、兽皮不足、食物不足、工具磨损/缺失、神龛已建成、烟熏房已建成等）
- 每个 NPC 的技能集 / 状态 / 性格 / 目标 / 今日记忆 / 约定

### 7.2 LLM 输出结构

```json
{
  "actions": [
    { "id": "oskar", "loc": "ruins", "skill": "build", "brief": "继续修屋顶" }
  ],
  "talks": [
    { "who": "agnes", "say": "今晚我炖一锅，你们都来。" }
  ],
  "commitments": [
    { "npc": "roderic", "day": 2, "hour": 8, "text": "跟 Oskar 一起去采石", "with": "oskar" }
  ]
}
```

### 7.3 规则引擎降级

LLM 不可用（无 API key 或超时）时，`BehaviorSystem._decide` 会按以下顺序走规则：

1. 体力极低 → 强制休息
2. 饥饿 → 尝试进食 / 烹饪
3. 夜晚 → 回屋睡觉（优先自己的民居）
4. 否则按优先级尝试 PRODUCTIVE 技能（build / hunt / chop_wood / cook …）
5. 最后兜底 rest

`_tryBuildProject` 在规则路径下会优先修废屋，其次选当前地点（若是建筑工地），再选已备料的项目，最后兜底第一个未完成项目。

### 7.4 心智整理（MindSystem）

每天 22:00 对每个 NPC 单独调用一次 LLM（`buildConsolidationPrompt`），将当天的记忆浓缩为：

- `daySummary`：一句话自述
- `shortTermGoal`：接下来几天想做的事
- `longTermGoal`：长远目标
- `commitments`：明确约定的未来事件

这些字段回写到 NPC，第二天的 Group Prompt 会带上它们，形成跨天的叙事连续性。`config.MIND` 控制上限。

## 8. UI

### 8.1 主界面

- **顶栏**：天数、小时、冬季阶段、暂停/加速、API 状态
- **左侧**：3 个 NPC 卡片（姓名可点击打开角色面板、状态条、当前行为）
- **中间**：10 个地点卡片（grid 布局，未建成的建筑以虚线+半透明呈现）
- **右侧**：日志 / 时间线回放

### 8.2 角色面板（点击 NPC 姓名触发）

显示该 NPC 的：
- 生存状态条（饱食 / 体力）
- 近期目标 / 远期目标
- 按 PHYSICAL/SOCIAL/MENTAL/RESTORE 分组的技能标签
- 人物背景故事

点击面板外任意位置关闭。

## 9. 配置中心（config.js）

所有可调数值集中在 `src/game/config.js`，避免数字散落在逻辑代码中：

| 常量组 | 控制内容 |
|---|---|
| TIME | 每日小时数、tick 毫秒、夜间时段、速度档位 |
| INITIAL_STATS | NPC 开局饱食/体力 |
| NEEDS | 每小时饱食/体力消耗 |
| ENV_TEMPERATURE | 冬季压力的倍率 |
| ENERGY_COST | 按行为类型的体力消耗 |
| RECOVERY | 睡眠/休息/进食恢复量 |
| THRESHOLD | 决策阈值（低体力/饥饿判定） |
| GATHER | 技能产出/消耗系数 |
| STARTING_ITEMS | 开局发放的物品 |
| LLM | LLM 参数（temperature、token 上限） |
| MIND | 记忆/约定/历史总结上限 |
| TOOL_RECIPES / REPAIR_COST / REPAIR_AMOUNT | 工具修理制作 |
| BUILDING_PROJECTS | 7 个建筑项目的材料/劳动/分配 |
| HOUSE_SLEEP_MULTIPLIER / WORKSHOP_REPAIR_AMOUNT / WORKSHOP_CRAFT_DISCOUNT | 建筑加成 |
