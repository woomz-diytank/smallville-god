# Smallville God 设计文档

## 1. 项目概述

中世纪欧洲背景下，8 名移民在荒废小镇的基础上合作生活、抵御冬天寒潮的模拟项目。纯前端运行，通过 LLM（Gemini）驱动 NPC 的叙事决策。

### 1.1 技术栈

- 构建：Vite 5（`npm run dev/build/preview`）
- 运行：浏览器端 ES Modules，无后端
- 渲染：Canvas 2D + DOM UI
- LLM：`@google/genai`（Gemini）

## 2. 架构哲学

### 2.1 借鉴 Pi 框架

Pi（pi-mono）是一个极简 AI Agent 框架，核心设计：

- **事件总线**：所有状态变化转化为事件
- **状态机**：Agent 生命周期为 idle → streaming → idle
- **极简工具集**：仅 4 个原子工具覆盖全部需求
- **事件驱动 Tick**：仅在收到事件时唤醒

### 2.2 对 NPC 系统的映射

| Pi 概念 | NPC 系统对应 |
|---------|------------|
| 事件总线 | NPC 的感知来源——环境变化、他人行为皆为事件 |
| 状态机 | 每个 NPC 拥有独立的认知状态机（纯逻辑，不需要 LLM） |
| 工具执行 | NPC 的行动——移动、制作、交谈都是"工具" |
| 记忆过滤 | 只把相关记忆发给 LLM，而不是全部历史 |

### 2.3 多 NPC 模拟策略：按地点动态分组（方案 C）

以 8 个 NPC 为例，处于同一地点的 NPC 分到同一个 LLM 会话（每组约 2 人），并行发起多个会话。

核心优势：共享上下文的粒度与物理空间对齐——同地点的 NPC 天然需要互相感知，不同地点的 NPC 天然信息隔离。

在 1 小时/Tick 的粒度下，LLM 以"导演"身份编排因果链（观察→对话→冲突→反应），而非让每个 NPC 独立决定一个原子动作。

## 3. NPC 行为框架

### 3.1 四层架构

```
层级1：需求系统（WHY）——NPC 此刻最想满足什么需求
         ↓ 驱动选择
层级2：行为库（WHAT）——NPC 从中挑选一个具体行为
         ↓ 约束于
层级3：能力集（CAN）——每个 NPC 拥有的技能子集
         ↓ 约束于
层级4：环境（WHERE）——当前地点 + 手中工具决定哪些行为可执行
```

### 3.2 需求层：SURVIVE / PRODUCE / BOND / SEEK

从两个正交维度推导出四个基本需求：

```
                物质            精神
           ┌───────────┬───────────┐
  指向自身 │  SURVIVE   │   SEEK    │
           │  存续      │   求知    │
           ├───────────┼───────────┤
  指向外部 │  PRODUCE   │   BOND    │
           │  创造      │   联结    │
           └───────────┴───────────┘
```

- **SURVIVE（存续）**：维持生存。吃饭、睡觉、避险、取暖。
- **PRODUCE（创造）**：产出成果。建造、制作、采集、烹饪。
- **BOND（联结）**：与他人建立关系。交谈、合作、领导。
- **SEEK（求知）**：理解世界。观察、学习、反思、祈祷。

同一个行为可以由不同需求驱动——Aldric 修工具是 PRODUCE 驱动，Erik 帮忙搬运是 BOND 驱动（渴望被接纳）。每个 NPC 的需求权重配比构成其独特的行为倾向。

### 3.3 行为层：PHYSICAL / SOCIAL / MENTAL / RESTORE

行为按与世界交互的客观方式分为四类：

- **PHYSICAL（劳作）**：身体作用于物质世界。采集、狩猎、建造、锻造、烹饪。
- **SOCIAL（交际）**：与他人的双向交互。交易、协调、安慰、教导。
- **MENTAL（思索）**：内在的信息处理。观察、祈祷、筹划、诊断。
- **RESTORE（休养）**：自我恢复。睡觉、吃饭、取暖、休息。

### 3.4 能力集设计原则

**NPC 不是由职业定义的，而是由其掌握的技能集合定义的。** 基本生存技能（采集、看火等）广泛分布于大多数成年人中；专精技能（锻造、狩猎、缝制等）仅少数人掌握。角色的独特性来自其技能组合 + 性格 + 需求权重，而不是一个职业标签。

技能普及度：

- 普遍（8/8人）：采集、看火
- 常见（4-5人）：伐木、搬运、搜寻、烹饪
- 少见（3人）：采石、修建
- 专精（仅1人）：锻造、狩猎、设陷阱、缝制衣物

真正的瓶颈往往不是"谁会做"，而是"工具在谁手上"——4 个人都能砍树，但可能只有一把斧头。

### 3.5 学习机制

所有 NPC 天生拥有两个学习技能：

- `practice`（练习）：PHYSICAL——动手练习不熟练的技能
- `learn_from`（请教）：SOCIAL——向掌握技能的人学习

学习动机由需求层驱动。系统维护学习记录，由 LLM 根据叙事合理性判断何时"学会"。习得后新技能加入能力集，行为边界永久扩展。

## 4. 物品系统

### 4.1 核心规则：物品来源对应行为

每件物品的来源必须对应行为库中的具体行为，分三类：

1. **采集**（从自然获取）：forage → 浆果/茅草；hunt → 生肉/兽皮；chop_wood → 木材 ...
2. **制作**（从原料加工）：cook → 熟肉；craft_clothing → 皮毛外套；smith → 工具 ...
3. **交易**（从外界换取）：trade → 针线/毛毯/绳套陷阱 ...

### 4.2 物品分类

**消耗品（8种）**：浆果、生肉、熟肉、木材、石块、兽皮、茅草、废铁

**耐久品（11种）**：斧头、铁锤、猎刀、针线、铁锅、锯子、弓、绳套陷阱、皮毛外套、皮靴、毛毯

### 4.3 资源链

```
采集链：
  森林 ──forage──→ 浆果、茅草
  森林 + 斧头 ──chop_wood──→ 木材
  森林 + 弓 ──hunt──→ 生肉、兽皮
  森林 + 陷阱 ──trap──→ 生肉、兽皮
  废墟 + 铁锤 ──gather_stone──→ 石块
  废墟 ──scavenge──→ 废铁

制作链：
  生肉 + 木材 + 铁锅 ──cook──→ 熟肉
  兽皮 + 针线 ──craft_clothing──→ 皮毛外套 / 皮靴
  废铁 + 木材 + 铁锤 ──smith──→ 工具 / 武器
  木材 + 石块 + 茅草 + 铁锤 + 锯子 ──build──→ 房屋修缮

交易链：
  货郎定期到访 → 用兽皮等换取针线、毛毯、陷阱、废铁
```

### 4.4 耐久度系统

耐久品有 `maxDurability` 值，使用时递减。降为 0 时物品损坏不可用。`smith` 技能可修复工具和武器。

### 4.5 物品存储

- NPC 个人背包：携带的工具、武器、衣物
- 地点公共存储：每个地点的共享物资
- `haul` 技能在地点间搬运物资

### 4.6 实现

- **ItemRegistry**（`src/game/data/ItemRegistry.js`）：从 JSON 加载物品定义，按 id/category 索引查询
- **ItemSystem**（`src/game/systems/ItemSystem.js`）：运行时库存管理、技能-物品校验、物品转移

## 5. LLM 决策架构

### 5.1 核心原则

LLM 是主要决策者，规则引擎是校验者。行为框架的所有组件对 LLM 而言是结构化的 prompt 上下文：

```
需求权重   → 告诉 LLM "这个人在乎什么"
行为分类   → 约束 LLM 输出的结构格式
能力集     → 告诉 LLM "这个人会做哪些事"
物品状态   → 告诉 LLM "手上有什么工具、需要什么材料"
地点兼容性 → 告诉 LLM "在这个地点能做什么"
```

### 5.2 LLM 输出结构

```javascript
{
  hour: 14,
  location: 'forest',
  need: 'SURVIVE',
  type: 'PHYSICAL',
  skill: 'hunt',
  detail: '沿着溪边小路寻找猎物踪迹',
  thought: '今天必须抓到点什么，不然晚饭没着落',
  itemsUsed: ['bow'],
  itemsProduced: ['raw_meat', 'hide'],
  needShift: { SEEK: +0.05 },
  skillAcquired: null
}
```

### 5.3 规则引擎校验

LLM 输出后，规则引擎校验：

- 使用的技能是否在 NPC 的能力集中
- 行为是否与当前地点兼容
- 所需工具是否在手（物品系统校验）
- 所需材料是否足够（物品系统校验）
- 校验不通过时：要求 LLM 重新选择，或 fallback 到规则生成

### 5.4 降级方案

LLM 不可用时，系统退化为规则引擎驱动：按需求权重随机选择行为类型，按能力集 ∩ 地点 ∩ 物品可用性筛选技能，随机执行。

## 6. 地图与地点

6 个地点：广场、废弃教堂、旧铁匠铺、废屋区、森林、储藏室。

每个地点限制可用技能，NPC 的实际可用行为 = 自身技能 ∩ 地点兼容技能 ∩ 物品可用性。

## 7. 时间与季节

模拟跨度约 12 周（秋末至来年初春），分四个阶段：

| 阶段 | 周数 | 生存压力 | 特征 |
|------|------|---------|------|
| 秋末 | 1-3 | 低 | 资源充足，备冬窗口 |
| 深秋 | 4-6 | 中 | 气温骤降，备冬紧迫 |
| 初冬 | 7-9 | 高 | 大雪封路，协作变得必要 |
| 隆冬 | 10-12 | 极端 | 暴风雪、食物见底、疾病风险 |

## 8. 交易系统

货郎每 2-3 周到访广场。居民用剩余物资（兽皮、浆果、熟肉等）换取本地无法生产的物品（针线、毛毯、绳套陷阱等）。交易由 LLM 叙事，规则引擎校验物品数量。
